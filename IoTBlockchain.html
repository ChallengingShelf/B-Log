Using Ethereum Blockchain to Authenticate IoT Node Transaction: What I Found Out

Themes: Blockchain | IoT Nodes | Network Transactions | Sissyphus' boulder

Daniel Hamilton

What this was about

I investigated whether Ethereum blockchain technology could be used to authenticate communications between IoT devices in a simulated network. The idea was straightforward: instead of relying on weak or absent security measures on low-powered IoT devices, use a smart contract on the Ethereum blockchain as a gatekeeper. Nodes register their public keys on the blockchain, and before any two nodes communicate, the smart contract verifies both are registered and legitimate. To my knowledge, this hadn't been done before in quite this way, which made it interesting ŌĆö and at times, very difficult.

Why IoT Security is a problem worth solving

IoT is everywhere. Smart speakers, video doorbells, thermostats, wearables ŌĆö there are projected to be 55.7 billion IoT devices worldwide by 2025*, generating 80 zettabytes of data. These devices are used in homes, hospitals, supply chains, and increasingly in critical infrastructure. The security situation is bad. Here's why:

  • IoT devices are low-powered. They often can't run encryption because the computational cost drains their batteries. So often communications are unencrypted.
  • Weak default passwords ship with devices and rarely get changed.
  • Software goes unpatched for months or years, sometimes indefinitely, because devices can't afford downtime.
  • No standardisation exists. Every IoT network is different, with different hardware, software, and protocols. There's no blanket security approach that works across the board.
  • Centralised data structures create single points of failure. If that server goes down ŌĆö or gets attacked ŌĆö the whole network is affected.

Traditional cybersecurity tools like TLS and PKI certificates require too much processing power for most IoT devices. Lightweight encryption alternatives exist but are weak enough that a determined attacker can break them. The field is stuck.

Note: This blog post is adapted from a 25,000 word research paper I conducted in 2023. Statistics will have changed, details will have been dropped. Don't trust everything you read on the internet.

The Proposed Solution

Jarvis, initiaze buzzword maximisation sequence.

My proposal: use the Ethereum blockchain and a smart contract to handle node authentication. The mechanism works like this:

  • Each IoT node has a public key (unique identifier).
  • When a node joins the network, it registers its public key with a smart contract deployed on the Ethereum blockchain.
  • Before any two nodes communicate, the smart contract checks whether both nodes have registered public keys.
  • If both are registered, communication is authenticated and proceeds. If not, it's rejected. Because blockchain records are famously immutable ŌĆö they can't be altered once written ŌĆö a node's identity can't be faked. A malicious node that hasn't registered simply can't communicate with the rest of the network.

The smart contract (`smartContractDan`) was written in Solidity and deployed on the Goerli Ethereum test network via Remix IDE. A MetaMask wallet was configured to cover transaction costs and sign interactions with the blockchain.

That's pretty neat huh, if it works.

Tech Stack

What's in use here?

  • Contiki OS / Cooja simulator ŌĆö used to simulate the IoT network with Sky mote nodes running `hello-world.c` firmware. Very simple.
  • Ubuntu 22.10 ŌĆö required because older versions of Ubuntu (bundled with Contiki 3.0) don't support Web3
  • Solidity / Remix IDE ŌĆö for writing and deploying the smart contract
  • MetaMask ŌĆö browser wallet used to deploy the contract and sign transactions
  • MQTT (Message Queuing Telemetry Transport) ŌĆö protocol used to connect the C firmware with the JavaScript function calling the smart contract
  • Paho MQTT C client library ŌĆö to enable MQTT within the Contiki C firmware

What Actually Happened

I'll be direct: the full implementation didn't work. The smart contract was written, compiled, and successfully deployed on the Goerli test network. MetaMask was configured. The `hello-world.c` firmware was modified to include MQTT functionality. The JavaScript file was written to interface with the smart contract. All the individual components were built.

The problem was the Cooja simulator itself. After multiple attempts to get the full pipeline running ŌĆö including reinstalling Ubuntu and Contiki from scratch several times ŌĆö a fatal build directory error in Cooja (`fatal: Not a valid object name HEAD`) prevented simulations from running. The JavaScript integration also remained incomplete; the link between the C firmware and the JavaScript file via MQTT was written but couldn't be evidenced as functional. This meant I couldn't generate the comparison data I intended to: power consumption before versus after blockchain authentication, packet delivery ratios, and latency figures.

Okay, still, what does the baseline show and what does related research say?

Baseline

Before the blockchain integration was attempted, the basic Cooja simulation ran successfully. Three Sky mote nodes ran the `hello-world.c` firmware for 15 minutes. The average power consumption across the network was 1.432 milliwatts (mW). This baseline is useful context: it's the starting point against which any increase from blockchain authentication would be measured. It was expected ŌĆö and is supported by related literature ŌĆö that adding blockchain authentication would significantly increase power consumption. This is the core problem with applying blockchain to IoT. The computational overhead of interacting with a blockchain per-communication is non-trivial for devices designed to sip power over months or years.

Related Research

At the time of writing, there aren't many papers exploring exactly this topic ŌĆö blockchain authentication of node-to-node communication in IoT simulations. The few that exist reach similar conclusions:

  • Blockchain can provide trust, integrity, and authentication in IoT environments.
  • It is likely not feasible in current IoT deployments because of resource-constrained, low-powered devices.
  • IoT devices would need significantly more computing power and battery capacity before this becomes practical at scale.

Yeah, that's fair enough, that's the expected result. The concept is sound but the hardware isn't there yet. Any maybe it won't be. It is a juxtaposition to be abusing IoT devices like this and expecting them to run highly resource consumptive process when they are simply not designed to by their nature. I'm okay with that.

The Cost Problem

There's also a financial reality to acknowledge. Every interaction with a smart contract on the Ethereum network costs Ether (gas fees). Deploying the contract costs a small amount. Every node registration costs a small amount. Every authentication check costs a small amount. In a small network this is manageable. In a network with hundreds or thousands of nodes all constantly authenticating communications, costs accumulate fast. For organisations that already underspend on cybersecurity, this is a hard sell.

The Goerli test network was used throughout to avoid real costs, but any real-world deployment would require the Ethereum mainnet.

Honest Reflections and next steps for research

Despite not producing simulation results, this paper does achieve its goals to an extent. Provides a working, deployed smart contract for IoT node authentication (source code included, verifiable on Etherscan). It also documents the full methodology: network setup, wallet configuration, contract deployment, and MQTT integration attempt. This establishes a proof of concept backed by extensive literature. Identifies the specific technical blockers ŌĆö Cooja compatibility issues, MQTT broker configuration complexity ŌĆö that future researchers need to solve. Also this sets a baseline power consumption reading for comparison in future work

Strengths & Weaknesses of the project

Strengths

  • Immutable identity records on the blockchain mean node identities genuinely cannot be faked
  • Decentralised authentication removes the single point of failure problem
  • The method is theoretically scalable. Theoretically.
  • Uses proven, existing technologies (Ethereum, Solidity, Web3)

Weaknesses

  • Power consumption will increase ŌĆö by how much remains unquantified because the simulation failed
  • Transaction costs make this impractical at scale on the Ethereum mainnet
  • The MQTT communications between node and JavaScript file are themselves unencrypted, which introduces its own vulnerability.
  • IoT authentication is not currently a priority for many organisations; lightweight encryption is considered good enough by most

Next Steps for Research

The two conditions that need to change before this becomes viable.

Better IoT batteries

More processing headroom means devices can handle the computational overhead of blockchain interactions.

Standardisation.

If an organisation like NIST were to standardise a blockchain-based authentication approach for IoT, it would force device manufacturers to build it in from the start. Future research should run this on real IoT hardware rather than Cooja, resolve the MQTT broker configuration, and produce actual power consumption comparison data. The methodology is here. The smart contract is deployed and verifiable.
